Use Cases

适合哪些 Amazon Flat File 上传问题？

Decode Processing Report errors适合批量上传后只看到 upload failed，需要先按 SKU、行号、字段和错误码生成修复清单的卖家。 Triage broken parent-child variations适合 size、color、style、material、pack 变体无法归组、dropdown 不显示或子体被拆散的场景。 Prepare a client-ready catalog repair report适合 Amazon VA、agency 和 catalog team 先把客户问题分层，再决定是否生成修正版 flat file。 Separate file-fixable errors from catalog conflicts适合已经试过 Seller Central / Variation Wizard，但仍不知道是文件问题还是 ASIN catalog 冲突的卖家。

Workflow

建议修复顺序

  1. 先处理高风险错误 例如 ASIN 冲突、变体关系错误、变体关系限制，这类可能阻断父子体创建或更新。
  2. 再补必填字段 缺少品牌、标题、分类属性、图片等字段时，先补齐模板要求，再重新上传。
  3. 最后处理格式问题 字段长度、枚举值、URL、大小写、空格等问题可以批量修正后统一重传。

Privacy

隐私说明

当前版本只在浏览器中读取 Processing Report 和可选 flat file 内容，不会把这些文件发送到本站服务器。页面可能加载广告和统计脚本；这些脚本不接收你的报告或 flat file 内容。刷新页面后，解析结果会被清空。你也可以先使用页面内置示例或下载 /samples 目录中的示例文件验证解析流程。

FAQ

Amazon Processing Report 常见问题

为什么要先看 Processing Report？

Seller Central 页面只会提示上传失败，真正的原因通常在 Processing Report 里。先定位错误码、SKU、行号和字段名，再改 Flat File，能减少反复上传。

工具能替代 Seller Support 吗？

不能。这个工具用于整理错误原因和修复步骤；如果目录 ASIN 信息、品牌权限或系统锁定导致无法修改，仍然需要联系 Amazon Seller Support。

为什么不直接上传 Excel 文件？

第一版优先支持 Processing Report 常见的 txt、csv、tsv 格式，避免把复杂 Excel 模板上传到服务器。后续会根据真实反馈决定是否增加 xlsx 解析。
